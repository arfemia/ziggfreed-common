# Changelog - Ziggfreed Common

The dev changelog for the shared, mod-agnostic Hytale primitive library. Newest first. No em-dashes.

## 1.1.0
A UI-theming + in-page-feedback + world-atmosphere primitives release, all additive over 1.0.0. Adds the generic retint engine (`ui/UiRetint`) + the mod-agnostic theme value model (`ui/theme/Palette` + `ThemeRecord`) a consumer's themeable menu builds on (with the inline recolor copies in the toast / dialogue / leaderboard / party pages converged onto the one primitive), the `ui/ZigRichButton` clickable-rich-text primitive, per-kind toast SFX plus a dialogue quest-completion toast hook, and world time-of-day + forced-weather control (`world/AtmosphereService`) + per-player forced music (`world/ForcedMusicService`).

- New: toasts play a per-kind 3D SFX with zero per-call wiring. `ToastKind` carries a default base-game `SoundEvent` id (SUCCESS / ERROR / WARNING chime on show; REWARD + INFO default to SILENT on purpose - a REWARD toast is the visual companion to a grant whose authority already plays the reward jingle, and INFO is passive), `ToastSpec` gains `withSound(id)` (override) + `silent()` (suppress) + `effectiveSoundId()`, and `ToastablePage.showToast` plays it at the player via `Sound3D` (world-thread, fully guarded - a missing asset degrades to silence). So a consumer's in-menu action feedback makes a sensible noise by severity without each call site wiring audio; a moment whose sound is owned elsewhere (a claim that plays its own cue) uses `silent()`.
- New: a dialogue can float an in-menu completion toast. `DialoguePage` now extends `ui/toast/ToastablePage` (its toast overlay is inert for every dialogue that does not use one), `DialogueActionExecutor.Outcome` carries an optional `completedId`, and `DialoguePageDeps` takes an optional `completionToast` hook (`completedId -> ToastSpec`, the 6-arg constructor still works and disables it). A consumer's action handler calls `out.reportCompleted(id)` when a turn-in / force-complete actually transitions some domain thing (a quest, an objective) to completed, and the page shows the consumer-built completion toast before reopening. Generic + domain-free: the id is opaque, the engine never builds the toast, the consumer supplies it.
- New: `ui/ZigRichButton` - the CLICKABLE rich-text / parameterized-text button primitive. A vanilla `Button` element wrapping a child `Label #Label` whose text is set on `.TextSpans`, so colour / bold / italic markup AND `{0}/{1}` parameter substitution and `Msg.join` glue render on the button - what a `TextButton.Text` cannot do (it prints the tags and `{0}` literally). Carries `text` / `color` / `iconStart` / `iconEnd` helpers over the shared tintable `@ZigRichRowBtnStyle` + `@ZigRichPrimaryBtnStyle` styles and the reusable `@ZigRichBtn` component in `ZigButtons.ui`, so a consumer can show a clickable row with styled / parameterized label text instead of a flat String button.
- Fix: in-menu toasts never auto-dismissed after a page reopened as a NEW instance. The auto-dismiss hide was bound to the page instance that scheduled it, but the dominant reopen flow is `showToast(...)` then `openCustomPage(ref, store, new SomePage(...))` - a different instance - so `onDismiss` set the scheduling instance's guard and the new instance never reset it, leaving the hide skipped forever (the toast stuck until the next page action). `ToastablePage` now keys the auto-dismiss on the player's CURRENTLY-ACTIVE toastable page (a static registry `renderToastInto` registers and `onDismiss` compare-removes), so a hide scheduled by any instance fires through whichever toastable page is open when it lands. The world-thread push rechecks active-membership + `dismissed` + `ref.isValid` so a delayed hide racing a navigation is dropped, never crashed. An earlier pass in this cycle only covered the self-reopen (`this`) case; this covers the new-instance reopen case a consumer's pages use.

- New: `ui/theme/Palette` + `ui/theme/ThemeRecord` - the generic, mod-agnostic theme VALUE MODEL that pairs with `ui/UiRetint`. `Palette` is a pure-data colour-slot set (3 base slots + named structural / button / text slots + an optional bespoke 9-slice `textureDir` swap set with per-texture borders; `hasTextures()` + a base 3-colour constructor that defaults the 9-slice insets). `ThemeRecord` carries one loaded theme's `id` / `displayNameKey` / `Palette` / a neutral `premium` tag / sort `order`, foldable over the `defaults < pack < owner` backbone (`AbstractKeyedAssetConfig`). Both are license-free: NO gate, NO entitlement, NO consumer policy - `premium` is a tag the consumer interprets (e.g. "requires an entitlement to paint"); the type carries no concept of "unlocked". Lifted out of a consumer's theme-config inner classes, so the consumer keeps only the gate / active-selection / persistence policy and its own theme content while the value model is shared.
- New: `ui/UiRetint` - the generic, mod-agnostic palette-to-selector retint primitive (the runtime `PatchStyle` recolor / texture-swap mechanism several consumers had each re-derived inline). `retintColor(cmd, sel, hex)` sets `.Background.Color` (the confirmed in-place 9-slice recolor); `retintButtonStates(cmd, elementSel, def, hov, pre)` sets a LIVE button element's per-state `.Style.{Default,Hovered,Pressed}.Background.Color` (the proven dialogue-option-row technique); `retintStyleStates(cmd, "@StyleName", ...)` is the best-guess named-style-definition form (one push reskins every button on a shared style, in-game-verify); `swapPatch(cmd, sel, texturePath, border, tint)` sends a typed `PatchStyle` texture swap via `setObject` (the runtime texture-PATH form is the one in-game-verify nuance); `isSixDigitHex` is the shared `#rrggbb` guard (each colour push is skipped for a null / non-hex value, never a bare-String `.Background`). It takes RESOLVED hex + selector params only - no theme model, no palette type, no gate, no consumer policy. The inline copies converge onto it: `ToastRenderer.apply` (the toast frame tint), `DialoguePage.applyOptionStyle` (the per-option button tint), `LeaderboardPage.style` + `PartyInvitePage.paintActiveTab` (the active-tab tint, keeping their label-style sets inline). A consumer's theme-retint helper is now a thin policy wrapper over it (mirroring how a consumer's page extends common's `ToastablePage`). Authority: `.claude/research/hytale-ui-runtime-patchstyle-textures.md`.
- New: `world/ForcedMusicService.applyFor(store, ref, containerId)` / `clearFor(store, ref)` - force a `MusicContainer` music bed onto one player by id. Resolves + validates the id via `MusicContainer.getAssetMap().getIndex`, sets the player's `ForcedMusicTracker` index, and pushes the `UpdateForcedMusic` packet directly to the player's handler (works whether or not the engine's `ForcedMusicSystems.Tick` runs in an instance world). `applyFor` returns false on a bad / unresolved id or invalid ref so the caller can retry (never latches index 0); `clearFor` sends index 0 to restore the default bed. World-thread, fully try-guarded. Lifted the MECHANISM from Kweebec Nightmare's `MusicBedService`; the dread-container candidate list + per-tier selection policy stay in the consumer, which iterates its own roster and calls `applyFor` per player.
- New: `world/AtmosphereService.setDayTime(world, dayTimeFraction, pauseTime)` / `setForcedWeather(world, weatherId)` / `lock(world, dayTimeFraction, weatherId)` - control a world's time-of-day and forced weather. `setDayTime` pins the clock (`0.0` = midnight / darkest, `0.5` = noon) and optionally pauses it; `setForcedWeather` validates the id (an unknown id would blank the sky, so it is skipped) before forcing both the live `WeatherResource` and the persisted `WorldConfig`, and a `null` id clears the override; `lock` composes a paused day-time set with a forced weather (the frozen-dark-midnight convenience). Each call self-hops via `world.execute` and is fully try-guarded. Lifted the MECHANISM from Kweebec Nightmare's `AtmosphereService`; the dark-weather candidate list + choice stay in the consumer.

## 1.0.0
First stable release. The static primitives, the branching NPC dialogue engine, and the full co-op instance + encounter framework are feature-complete for the 1.0 consumer surface (Kweebec Nightmare 1.0.0 is the exemplar consumer). This cycle adds runtime boss MAX-health scaling, per-encounter boss world-map marker knobs, and same-identity reward merging.

- New: `health/HealthUtil.scaleMaxHealth(store, ref, factor, key)` - raise an entity's Health-stat MAX by a multiplicative modifier (keyed for idempotency) and heal it to the new max. Idempotent: a no-op when the modifier is already present, the stat is not yet balanced, `factor` is 1.0, or the ref is invalid; world-thread, fully try-guarded. Lets a consumer scale a boss's HP by party size / difficulty at runtime without re-authoring the role. Kweebec Nightmare's Warden is the exemplar (party-size x per-difficulty scaling, applied on the first tick after spawn, re-applied once per phase entity).
- New: per-encounter world-map MARKER knobs on `instance/encounter/MultiPhaseBossAsset` (the reusable multi-phase boss) - `HealthPerPlayer` (the per-additional-player MAX-health fraction the consumer composes with its own difficulty multiplier; default 0.0 = no per-player scaling), `MarkerIcon` (the world-map POI texture id; default `Home.png`), and `MarkerUpdateSeconds` (the boss-follow re-place throttle; default 3s). All optional and default off / neutral, so existing bosses are unchanged. The consumer drops + moves the POI through the existing `worldmap/WorldMapMarkers`.
- New: `instance/reward/InstanceReward.merge(rewards)` - combine same-identity rewards (matching `kind` + `id` + `displayKey`) into one entry each, summing quantities, so a rolled loot table renders "x9 Moonbloom" ONCE instead of several repeated chips (and the claim store grants one merged entry instead of many). `COMMAND` entries never merge (each is a distinct side effect); first-seen order is preserved. Unit-tested (`InstanceRewardMergeTest`). Any consumer rolling a loot table (which intentionally yields repeated picks) routes its result through here.

## 0.3.2
- New: score-tiered **loot tables** in `instance/reward/` - a reusable, pack-authorable reward-pool primitive. `LootTable` rolls a fixed `Guaranteed` list plus `clamp(Rolls + score/ScorePerBonusRoll, 0, MaxRolls)` weighted picks among `Pool` entries the player's run score unlocks (`LootEntry`: a weighted, `minScore`-gated, quantity-ranged template, compact spec `[w<n>] [s<n>] <kind> <id> <qty|min-max> [displayKey]` - a superset of `InstanceReward`'s). Deterministic per injected seed (mirrors `SpawnRoster`'s weighted pick), so "better loot for a better score" comes from BOTH premium entries unlocking AND more rolls. Pattern-A `LootTableAsset` + `LootTableConfig` (`Server/<Mod>/LootTables/`, registered by `FrameworkAssetRegistrar`, `defaults < pack < owner`); a preset references one by the new `InstancePresetAsset.RewardTableId`. Common ships ZERO content (all pool JSON is consumer-authored). Unit-tested (`LootEntryTest`/`LootTableTest`); `LootTableAsset.CODEC` guarded by `AssetCodecInitTest`. Kweebec Nightmare's chase end-game rewards are the exemplar consumer.
- New: `instance/zone/ZoneHoldTimer` - a reusable co-op "hold this zone" objective primitive. The required number of occupants must stay in a zone TOGETHER for a configured duration, and the hold RESETS the instant the group breaks; it latches complete once the continuous hold elapses. PURE + engine-free (the caller injects the live occupant/required counts and its own `nowMillis`, mirroring `EncounterDirector`), unit-tested, with `isHolding()`/`remainingSeconds(now)`/`progress(now)` HUD queries. The consumer owns WHO counts as an occupant, WHERE the zone is, and what completion does. Reusable for an extraction pad, a capture point, king-of-the-hill, or a ward-defense hold. Lifted from Kweebec Nightmare's Chase extraction platform (the exemplar consumer); a future dungeon / raid consumer is the target.
- New: per-phase HELPER-THROWABLE cluster knobs on `instance/encounter/MultiPhaseBossAsset` (the reusable multi-phase boss). A boss phase can now author harvestable resource clusters the consumer's controller places + replenishes around the boss, so survivors gather + throw them at a high-HP boss to keep it killable: `ThrowableClusterId` (an opaque id the consumer resolves - Kweebec treats it as a prefab key), `Phase{1,2,3}ThrowableCount`, `ThrowableRingRadius`, and `ThrowableRespawnSeconds` (the configurable regrow timer). All optional, default off (zero counts), so existing bosses are unchanged. Placement reuses the existing `SpawnPlacement` ring + `EncounterDirector` primitives caller-side; common stays pure data. Kweebec Nightmare's Warden is the exemplar consumer (the boss-phase Emberbloom supply).

## 0.3.1
- New: shared `@ZigTabBtnStyle` segmented-tab / filter button in `Common/UI/Custom/Common/ZigButtons.ui` - a center-aligned sibling of `@ZigOptionBtnStyle`. One style covers active and inactive tabs: the consuming page tints the active tab's per-state `.Background.Color` from Java (the same technique the dialogue page uses to paint an option row). Lets a consumer page (a minigame leaderboard's party-size tabs, a paged list's category chips) reuse the shared button look instead of defining its own.

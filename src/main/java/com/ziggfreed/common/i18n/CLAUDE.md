# i18n/ - client-resolved Message factory

Router for `com.ziggfreed.common.i18n`. Mod-agnostic localization primitive lifted from the MMO Skill Tree's own `i18n.Msg` (RPG Stations extraction, phase 1 leg 1); the MMO's copy stays untouched (no churn - an existing, working call site does not need to move).

- **[`Msg`](Msg.java)**: builds Hytale `Message` values the CLIENT resolves against the viewer's own locale - the server never reads, caches, or persists a per-player locale. `tr(prefix, key, args...)` binds `{0},{1},...` under a CALLER-SUPPLIED namespace prefix (this class carries no fixed namespace of its own, so several consumer mods share one implementation); `key(fullKey, args...)` skips prefixing for a fully-qualified / another mod's / native key; `trNamed`/`keyNamed` bind `{name}` tokens instead. `raw(text)` is untranslated data (numbers, ids, player names) - never a localized name. `join(...)` glues parts for DIRECT placement only (a bare join carries neither `rawText` nor `messageId`, so it renders blank if nested as a `tr`/`key` param); `cat(...)` is the param-safe composite, folding pairwise through a shipped locale-neutral `common.fmt.cat = "{0}{1}"` key so every fold node carries a `messageId` and resolves recursively even when nested. `bold(m)`/`color(m, hex)` are thin fluent wrappers over the engine `Message#bold`/`Message#color` (no import of the engine type needed at a call site that only wants styling).
- **The arg-binding invariant** (carried over from the MMO's own `i18n.Msg`, restated here since it is the entire correctness story): an arg that is ITSELF localized (a name, a nested template) MUST be passed as a nested `Message`, never pre-resolved to a String - and even a plain String arg is wrapped as a nested `Message.raw(...)`, not the bare `param(key, String)` overload (the client cannot deserialize a raw `StringParamValue` against a plain-String host property and disconnects).
- **A consumer wanting a prefix-free call site** wraps this with its own tiny facade (a one-line `Msg.tr("rpgstations." + key, args)` or a per-mod class), mirroring how the MMO's own `i18n.Msg` already reads.
- **[`GeneratedLangPack`](GeneratedLangPack.java)**: ship a RUNTIME-GENERATED localization overlay to clients - `writeLocaleFiles(root, filename, perLocale, header)` writes one `<root>/Server/Languages/<locale>/<filename>` per locale (idempotent overwrite, empty-value entries skipped), and `registerZipPack(root, zip, group, name, version, desc)` builds an IMMUTABLE zip from it and registers it RUNTIME via `AssetModule.registerPack` (unregister-then-register on re-invocation). The catalog-sync chain is `registerPack -> I18nModule.loadMessagesFromPack -> per-client sendTranslations at join`; keys land under the `filename`-minus-`.lang` PREFIX (`storedKeyPrefix`/`storedKey` are the pure authority a consumer's display + generator share so a lookup key never drifts from a generated entry key). **IMMUTABLE zip, never a watched directory** (the reason this primitive exists): a mutable dir pack's `AssetMonitor` PRUNES the entire matching vanilla namespace (all `server.*`/`client.*` keys) on any post-registration MODIFY; a zip is never watched. Intended use is boot-once generation (joining clients covered); the already-connected live push a mutable dir would add is not worth the catalog-prune hazard. Guarded (register failures return `false`, never throw). The MMO's item-description overlay + per-skill tooltip generators are the consumers; the reusable extract of the hand-rolled `StatTooltipLangGenerator` machinery.
- **Conventions**: `@Nonnull`/`@Nullable`; no em-dashes; rich text (colour/bold/`\n`) still renders ONLY on a `TextSpans` sink, never `.Text` - this factory builds the `Message`, it does not choose where it gets set.

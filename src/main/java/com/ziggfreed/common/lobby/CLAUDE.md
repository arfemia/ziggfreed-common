# lobby/ - generic fill-window + countdown matchmaking queue

Router for `com.ziggfreed.common.lobby`. A net-new, mod-agnostic multiplayer queue (no native Hytale party/lobby primitive exists). A consumer holds one [`LobbyService`](LobbyService.java) and feeds a snapshotted party to its own launcher; everything from "player asks to play" up to "party handed to the launcher" is generic, everything the launcher does (ruleset, instance world, teleport, round registry) is consumer policy.

- **THE DRY BOUNDARY: the queue imports no world/store/mod types.** Only `UUID`/`String`/`Message` + the functional seams. The launch is a [`RoundLauncher`](RoundLauncher.java) `BiFunction<UUID initiator, List<UUID> party> -> CompletableFuture<?>`; membership-truth is an injected `Predicate<UUID> alreadyEngaged` (the consumer's round registry, NEVER duplicated here); presence is a `Predicate<UUID> online` (defaults to a store-free `Universe.getPlayer` lookup, [`LobbyService.universeOnline`](LobbyService.java)); all player-facing text is a [`QueueMessages`](QueueMessages.java) provider of pre-built `Message`s (so common stays locale-free).
- **[`MatchmakingQueue`](MatchmakingQueue.java) is the state machine**: `OPEN -> COUNTDOWN -> LAUNCHING -> CLOSED` ([`QueueState`](QueueState.java)). `join` hitting `maxParty` -> countdown now; hitting the fill trigger (`minParty`, or 1 when `allowSolo`) -> fill window, which on expiry -> countdown; `leave` below the floor -> cancel back to OPEN; `forceStart` (initiator only, if `leaderForceStart`) skips the fill window. On countdown expiry the party is snapshotted, re-validated (online + `!alreadyEngaged`), the initiator re-picked to the first still-eligible joiner (abort if none), and the launcher fired. Knobs are a [`LobbyConfig`](LobbyConfig.java) value object.
- **[`LobbyService`](LobbyService.java) owns the live queues + the global single-queue-per-player reservation** (a `queuedTo` map mirroring a round registry's `playerToRound`, so a player is in &le;1 queue) + the one daemon scheduler. `queue(key, config, launcher, alreadyEngaged[, online], messages)` is get-or-create (a CLOSED queue is replaced). `isQueued`/`queueKeyOf`/`leave`/`shutdown` round it out. A consumer wraps it in a thin static accessor (kweebec's `KweebecLobby.queue(QueueKey)`).
- **`launchSolo(baseKey, uuid, config, launcher, alreadyEngaged, messages)`** is the SOLO "play now" path (the [`../../instance/play/`](../../instance/play/CLAUDE.md) Play screen's Solo card): a private self-scoped queue (`QueueKey.privateQueue(game, preset, uuid)`) + a forced `new LobbyConfig(1, maxParty, 0, 0, true, false)` fed to `queueParty`. The `fillTimeout<=0`/`countdown<=0` fast-paths fire `fireLaunch` immediately off-lock - no fill wait, no countdown, no strangers - while still honouring the one-queue-per-player reservation. Pure composition; no `MatchmakingQueue` change.
- **Threading = the `RoundStateMachine` pattern.** An owned single daemon scheduler ([`DelayScheduler`](DelayScheduler.java), wrapping `ScheduledExecutorService`) runs the fill window + per-second countdown. **HARD RULE: the tick is packet-only** - UUID iteration + `Universe.getPlayer` (store-free) + `Notify`/`EventTitles` packet writes; NEVER a Store/Ref read. The foreign `launcher.launch` call is made OFF the queue lock (the launcher owns any `world.execute` hop). A persistent HUD indicator would be world-thread-bound, so it is explicitly out of scope here.
- **`DelayScheduler` is also the test seam.** The state machine is deterministically unit-tested ([`MatchmakingQueueTest`](../../../../../../test/java/com/ziggfreed/common/lobby/MatchmakingQueueTest.java)) by injecting a manual scheduler and firing tasks by hand (no wall-clock waiting); delivery is a guarded no-op with no server. Covers fill/countdown/launch, max-skip, allowSolo, leave-cancel, double-join, engaged/offline filtering + initiator re-pick, launch abort, force-start, and CLOSED-queue refresh.
- **v1 UI = zero new `.ui`**: join/leave toasts + a "launching in N" `EventTitles` banner, all from `QueueMessages`. The preset PICKER is the existing [`dialogue/`](../dialogue/CLAUDE.md) page (a guide NPC), not a lobby page.
- **UI read seams + party group-join (the instance-experience layer).** A queue SCREEN renders ONLY a [`QueueSnapshot`](QueueSnapshot.java) (`snapshot()` / `members()` / `countdownRemaining()`, all built under the lock) and refreshes via a registered [`QueueListener`](QueueListener.java) (`onChange`/`onLaunch`, fired packet-only like the existing toasts - cheap, no Store/Ref). A pre-formed party queues AS A UNIT via [`MatchmakingQueue.joinParty`](MatchmakingQueue.java) / [`LobbyService.queueParty`](LobbyService.java) -> [`GroupJoinResult`](GroupJoinResult.java): all-or-none over the SAME `queuedTo` reservation (rolls back every reservation on any conflict), owner forced first so `fireLaunch` re-picks it as initiator (a PUBLIC queue with an earlier joiner keeps that joiner - correct backfill). PUBLIC vs PRIVATE is a [`QueueKey`](QueueKey.java) convention: `QueueKey.publicQueue` (shared, backfill with strangers) vs `QueueKey.privateQueue(gameId, presetId, partyId)` (scope = party id, launches that party alone). The party PRIMITIVE itself lives in [`../party/`](../party/CLAUDE.md); the lobby only learned to accept a group. Plan: [[proud-wondering-tiger]].

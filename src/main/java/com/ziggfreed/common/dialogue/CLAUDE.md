# dialogue/ - the generic branching NPC dialogue engine

Router for `com.ziggfreed.common.dialogue`. A self-contained, mod-agnostic dialogue runtime lifted from hyMMO (which is being re-integrated as its first EXTENDER). A consumer builds ONE [`DialogueEngine`](DialogueEngine.java) via the builder, supplies a few seams, and opens [`page/DialoguePage`](page/DialoguePage.java); the `.ui` ships once here and resolves client-side.

- **PARADIGM: per-consumer engine, NOT a shared mutable registry.** `DialogueEngine.builder()` pre-seeds the generic action/condition/sugar types, the consumer `.action(...)`/`.condition(...)`/`.router(...)`s its own, and `.build()` assembles the codec graph ONCE before any decode - so there is no registration race and one mod's domain types never leak into another's. The MMO registers quest/reward/gate types; a minigame builds with generics only.
- **[`DialogueAction`](DialogueAction.java) + [`DialogueCondition`](DialogueCondition.java) are BOTH `Type`-discriminated polymorphic lists** (`Conditions` is an array of `{Type:...}` objects, symmetric with `Actions`). Generic actions: `Goto`/`Close`/`SetFlag` + the carriers `Talk`/`OpenPage`. Generic conditions: `Flag`/`NotFlag` + the **boolean combinators `AllOf`/`AnyOf`/`Not`** (each holds a child `Conditions` list under `All`/`Any`/`Of`; assembled in `build()` after the shared `conditionsArray`, evaluated by delegating each child back through `conditionsPass`). The dispatch `CodecMapCodec`s are engine-instance-owned (assembled in [`DialogueEngine.Builder.build`](DialogueEngine.java)); each leaf subtype carries its own field `BuilderCodec` (PascalCase keys, guarded by [`DialogueEngineTest`](../../../../../../test/java/com/ziggfreed/common/dialogue/DialogueEngineTest.java)).
- **A `DialogueNode` can self-gate with its own `Conditions`** (the same shared list). When a `Start` candidate resolves to a node, `resolveEntryNodeId` ALSO requires the node's conditions to pass - so state-varied beats collapse into ONE node map with per-node runtime conditions instead of `(node x state)` duplication. A `DialogueOption` can carry a `Presentation` (nested `{Color, Icon:{Item,Glyph}}` codec): the page overrides the action-derived `DialogueOptionStyle` with a per-option colour (retinted hover/press) + an icon (a game item id via the `#OptIconItem` grid slot, or a fixed glyph token).
- **One registration binds schema + behavior + presentation + sugar so they can't drift**: a [`DialogueActionType`](DialogueActionType.java) carries the codec + a [`DialogueActionHandler`](DialogueActionHandler.java) + an optional [`DialogueOptionStyle`](DialogueOptionStyle.java) kind + an optional [`DialogueSugarExpander`](DialogueSugarExpander.java); a [`DialogueConditionType`](DialogueConditionType.java) carries the codec + a [`DialogueConditionEvaluator`](DialogueConditionEvaluator.java). The executor dispatches by a handler MAP (not `instanceof`), which is what lets a consumer add an action without editing the engine.
- **The opaque per-player handle is [`DialogueContext`](DialogueContext.java)** (`DialogueExecContext` adds dialogue/node/option). The engine only reads the generic flag store + forwards the context to registered evaluators/handlers; a consumer's evaluator reaches its domain state via `payload(Class)`. Build it with [`page/SimpleDialogueExecContext`](page/SimpleDialogueExecContext.java) inside a [`page/DialogueContextFactory`](page/DialogueContextFactory.java). Flag storage is the consumer's [`DialogueFlagStore`](DialogueFlagStore.java) (MMO: a player component; minigame: in-memory per-round).
- **Seams**: [`DialoguePageRouter`](DialoguePageRouter.java) (the `OpenPage` target -> open a page, default NONE), [`NpcNameProvider`](NpcNameProvider.java) (header name, default NONE), [`DialogueHeaderAnnotation`](DialogueHeaderAnnotation.java) (the one-line hint row, default NONE), [`i18n/DialogueI18n`](i18n/DialogueI18n.java) (namespace prefix + en-US `hasKey` probe; the ready-made [`I18nModuleDialogueI18n`](i18n/I18nModuleDialogueI18n.java) is zero-config). All bundled into a [`page/DialoguePageDeps`](page/DialoguePageDeps.java) built once at startup.
- **Reuse is NATIVE `Parent` inheritance (native-lite, 1.5.0), NOT a hand-rolled template DSL.** The old `extends`/`params`/`nodeOverrides`/`extraNodes`/`PruneIfEmpty` resolver (`template/`) was REMOVED. A body may carry a top-level `"Parent": "<id>"`; [`DialogueBodyResolver`](DialogueBodyResolver.java) resolves the chain (topological, memoized, cycle-guarded) and decodes each through [`DialogueEngine.decodeWithParent`](DialogueEngine.java) -> the engine's real `decodeAndInheritJson`. `Start`/`Nodes` + node fields are `appendInherited` (omit a field -> inherit the parent's); the `Nodes` map uses [`../codec/InheritMapCodec`](../codec/InheritMapCodec.java) so a child overrides/adds a node BY KEY while inheriting the rest (each `DialogueNode` is a `BuilderCodec` = `InheritCodec`, so its fields deep-merge too). Option `sugar` still runs as a pre-pass (`engine.sugar().desugar` inside the resolver); loc-keys are convention-derived (`dialogue.<id>.<node>.text`/`.opt.<i>`), so a child gets its own namespace for free. Params/prune are gone: state-selection is runtime node `Conditions` + `AnyOf`; per-NPC quest ids are literal field overrides.
- **[`validate/DialogueStructureValidator`](validate/DialogueStructureValidator.java)** is the STRUCTURAL audit only (missing/dangling start, dangling Goto, unreachable, empty-after-resolve) returning neutral `Issue`s; a consumer that registers domain actions runs its OWN pass for their refs and merges.
- **[`page/DialoguePage`](page/DialoguePage.java)** is the generic page (extends `ui/toast/ToastablePage`). Every exit path sends a response (the client spins otherwise). **In-menu completion toast (optional):** when an action reports a just-completed thing via `DialogueActionExecutor.Mut.reportCompleted(id)` (carried back on `Outcome.completedId`) AND the consumer wired a `completionToast` (`completedId -> ToastSpec`) into `DialoguePageDeps`, the page floats that toast over the dialogue before reopening. Engine-side this is domain-free: the id is opaque (a quest, an objective, anything), the executor never builds a toast, the consumer supplies it; the overlay is inert for every dialogue that wires neither. The `.ui` lives in `src/main/resources/Common/UI/Custom/` as `Pages/ZigDialoguePage.ui` + `Pages/ZigDialogueOptionRow.ui` + `Common/ZigButtons.ui` + `Common/ZigFrames.ui` (neutral `Zig*` names so they never collide with a consumer's own `.ui`); all textures are base-game. The `.ui` is NOT gradle-validated - smoke-test in-game.
- **RICH TEXT = native inline markup in the `.lang`, rendered on a Label's `TextSpans`.** Hytale's client parses `<color is="#rrggbb">..</color>`, `<b>..</b>`, `<i>..</i>` (and `\n`) inside a localized string when it is shown via a **Label's `TextSpans`** property - this is the engine's OWN mechanism (vanilla `items.*.description` + `customUI.*` strings use it across all locales, so it is per-locale + client-resolved). The node text is set on **`#NodeText.TextSpans`** (NOT `.Text` - `.Text` shows the tags LITERALLY, the shipped bug), so any markup the consumer writes in its `.lang` value just works, in every locale, with zero engine support. Markup may carry `{param}` too. Option rows are `TextButton`s - **buttons have NO `TextSpans`**, so an option label cannot be coloured by markup; its colour comes from [`DialogueOptionStyle`](DialogueOptionStyle.java). Toasts/event-titles + the item-icon/grid `name` tooltip are String-only sinks (no rich text); the item `description` tooltip IS rich. (A server-side markup parser + a per-node-param scheme were both tried and removed - native `.lang` markup on `TextSpans` is simpler and already per-locale.)

- **[`asset/`](asset/) - the pack-authored dialogue STORE layer (Pattern B, common-registered).** `ZcDialogueAsset` + `ZcDialogueTemplateAsset` (raw-`Payload` over `../asset/AbstractRawJsonAsset`) deliver bodies + templates from `Server/ZiggfreedCommon/Dialogues/` + `/DialogueTemplates/` (registered by `../asset/FrameworkAssetRegistrar`; the dialogues store `loadsAfter` the templates store so `extends` resolves). `DialogueAssetStore` holds the raw `id -> JsonObject` bodies (+ an `Owner` per body) + the template bodies and exposes `resolveAll(engine, ownerFilter)` -> `Map<String,NpcDialogue>` (runs `DialogueBodyResolver` over a POOL of templates + bodies, emitting only the owner-matched bodies - so a `DialogueTemplate` is now just a native `Parent` BASE that others inherit by id, not a DSL `extends` target). The decode is per-consumer-engine, NOT a static codec, so a consumer builds its engine first then calls `resolveAll` filtered to its own `Owner` after assets load. Lifted from the MMO's `DialogueConfig`; Kweebec is the first consumer (authored `Clash_Intro.json` + `Nightmares_Intro.json` with `Owner:"kweebec"`, both DSL-free so unchanged by the native-lite switch).

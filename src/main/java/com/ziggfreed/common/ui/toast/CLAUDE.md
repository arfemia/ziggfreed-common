# ui/toast/ - transport-agnostic in-page toast engine

Router for `com.ziggfreed.common.ui.toast`. A mod-agnostic, in-menu toast overlay lifted from the MMO Skill Tree's `ui/toast/` (the MMO is the shared-file owner and now re-derives its own `ToastablePage` as a thin subclass over this engine). i18n is parameterized OUT: the engine takes pre-built client-resolved `Message`s only - it imports no consumer `Msg`/reward renderer. Element ids/styles are strictly neutral (`#Zig*`) so the `.ui` never collides with a consumer's own pack when both jars are installed.

- **[`ToastSpec`](ToastSpec.java) + [`ToastLine`](ToastLine.java) + [`ToastKind`](ToastKind.java)**: the immutable, transport-agnostic model. Build via `ToastSpec.of(kind, message)` then `withLines/withDuration/withTitle/withIcon`; lines carry an optional item-icon id + a pre-built `Message` (the consumer maps its own reward/feedback rows in, never re-resolved here). `ToastKind` maps each severity to a panel text/frame colour + a native `NotificationStyle`.
- **[`ToastStore`](ToastStore.java) is the SINGLE current-toast authority, keyed by player UUID, held STATICALLY** - pages reopen as fresh instances, so per-instance toast state would be lost on the reopen meant to render it. A monotonic `GEN` token (`show` -> `clearIfCurrent(gen)`) stops a stale scheduled auto-dismiss from clearing a newer toast.
- **[`ToastController`](ToastController.java) is the engine; [`ToastSurface`](ToastSurface.java) is the ONLY transport seam** (a `@FunctionalInterface push(UICommandBuilder)`). `show(...)` writes the store + pushes the show command, then schedules the hide via `HytaleServer.SCHEDULED_EXECUTOR`; `renderInto(cmd)` repaints the live toast into a full `build()` so it survives a page reopen. Bind the surface to a page `sendUpdate` (no reopen, scroll preserved).
- **[`ToastRenderer`](ToastRenderer.java) is the SOLE owner of the `#ZigToast` element-id contract** over the shared `Pages/ZigToast.ui` fragment (+ `ZigToastFrame.png`, a white 9-slice frame tinted per kind). Show/hide is driven by `Anchor` collapse to `0x0` (a `sendUpdate` can set `Anchor` but NOT flip `Visible` false->true); `MAX_LINES` reward rows are pre-baked because `sendUpdate` can only restyle existing elements, never append. `MAX_LINES` MUST match the `#ZigToastRow<i>` count in the `.ui`.
- **[`ToastablePage`](ToastablePage.java) is the page base** a consumer page extends: `showToast(kind, msg)` / `showToast(spec)` set the toast (call after an in-page action, then reopen the page), `renderToastInto(cmd)` paints it as the LAST root sibling (call last in `build`), and the `onDismiss` guard stops a stale auto-dismiss from pushing into a replaced page. The MMO subclass adds its reward-list helpers (`showRewardToast` over its `RewardListRenderer`); common's own `instance/result/ResultsPage`, `party/page/PartyInvitePage`, `instance/queue/QueuePage` extend this for in-page reward/feedback toasts.

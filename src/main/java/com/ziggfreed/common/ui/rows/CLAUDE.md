# ui/rows/ - fixed-slot icon + rich-text LEDGER row primitive

Router for `com.ziggfreed.common.ui.rows`. A mod-agnostic, transport-agnostic ledger-row model + painter lifted from the MMO Skill Tree's session-summary HUD (the MMO is the shared-file owner; it keeps only its policy - skill-icon / item-name resolution, colour/bold conventions, session tallies -> rows - and consumes this primitive).

- **[`SummaryRow`](SummaryRow.java)**: the immutable value - an icon item id + an already-composed, already-styled client-resolved `Message` + a descriptive semantic `Kind` (`XP`/`CONSUMED`/`PRODUCED`/`LUCKY`/`ENHANCE`, generic reward-ledger roles; `ENHANCE` = an existing item modified in place, e.g. a stamp/durability outcome). All colour/bold is baked into the `Message` by the caller (`Message.color`/`Message.bold` composed via the consumer's own `Msg.cat`); this type only carries the result. `Kind` is metadata (ordering / per-kind icon fallback), NOT read by the renderer.
- **[`SummaryRowRenderer`](SummaryRowRenderer.java)**: paints a `List<SummaryRow>` into a FIXED set of pre-declared `.ui` slots the CONSUMER declares - `render(cmd, rowSelectorPrefix, maxSlots, rows)` addresses row `i` as `rowSelectorPrefix + i` (each expected to hold an `ItemGrid #Icon` + a `Label #Name`), hides unused/overflow slots, and RETURNS the overflow count for a "+N more" row (or `renderWithOverflow(...)` sets that label too). Never `cmd.append` - the fixed-slot HUD/page convention. Works over any `UICommandBuilder` sink (HUD partial update, page `sendUpdate`, full `build`).
- **Sibling to [`ui/toast/ToastLine`](../toast/ToastLine.java), NOT unified (deliberate design call, 1.5.0 lift).** Both are the same shape at heart (optional icon id + a client-resolved `Message`), but a `ToastLine` is a TRANSIENT overlay welded to the toast transport (its `ToastController`/`ToastStore` lifecycle, a `quantity` badge, a `ToastKind` that also drives per-kind SFX) while a `SummaryRow` is a PERSISTENT fixed-slot ledger row with a generic parameterized-selector renderer. Unifying would force toast transport semantics onto a plain ledger (or drag the fixed-slot painter into the toast model); keeping siblings lets a consumer pick the surface that fits without a third forked model. The results-strip [`instance/result/RewardChip`](../../instance/result/RewardChip.java) is a third, distinct shape again (it carries a `pending` state and renders as a strip, not a fixed-slot ledger).
- **Consumers**: the MMO session-summary HUD (skill-XP rows + item rows in one ledger). A minigame's rewards / results LEDGER can adopt it the same way (declare `#Prefix0..N` slots with `#Icon`/`#Name`, build `SummaryRow`s, call `render`); the results REWARD-STRIP (`ResultsPage` + `RewardChip`) is a separate surface today - adopting rows there is a rewards-UI design pass, not a mechanical swap.
- **Conventions**: engine `Message` only (no consumer `Msg`); rich text renders on a Label's `TextSpans` (never `.Text`); `@Nonnull`/`@Nullable`; no em-dashes.
